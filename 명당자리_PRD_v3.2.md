## 1. 문서 개요

### 1.1 작성 목적
디자인·개발·기획 팀 간 목표를 정렬하고, 실현 가능한 MVP 개발 범위를 확정한다.

### 1.2 핵심 가치 제안 (EVP)
"단순한 집 찾기를 넘어, 내 사주에 호응하는 공간과 배치를 찾아주는 라이프스타일 큐레이션."

### 1.3 문제 정의 (Problem Statement)
- **주거 의사결정의 불안.** 객관 지표(교통·학군·시세) 외에 "이 선택이 나에게 맞나"라는 **심리적 확신**을 채워주는 도구가 부족하다.
- **운세 서비스의 단발성.** 기존 사주/운세 앱은 결과를 한 번 보면 다시 올 이유가 약하다.
- **개운 인테리어의 정보 비대칭.** "어떤 색/소재가 나에게 맞는가"가 흩어져 있어 구매 결정으로 이어지기 어렵다.

### 1.4 포지셔닝 & 카테고리 정의 (중요)
본 서비스는 **라이프스타일·엔터테인먼트**로 포지셔닝한다. 무속적 톤을 배제하고, "운이 트인다/병이 낫는다"는 식의 **효능·결과 보장 표현을 전 영역에서 금지**한다(§11 참조).

## 2. 목표 & 비목표 (Goals & Non-Goals)

### 2.1 목표
1. 생년월일시 입력만으로 **오행 분석 → 지명/지형(행정동) 안내 → 침대 머리 방향 → 맞춤 상품**까지 한 흐름으로 제공.
2. "결과 1회 소비"를 넘어 **다시 찾는** 데일리 개운 루틴 확보.
3. 공유 가능한 결과 카드로 **바이럴 루프** 구축.
4. Phase 1 내 **커머스 아웃링크 전환** 데이터를 확보해 수익화 가설 검증.

### 2.2 비목표 (이번 범위 아님 — 의도적 제외)
- **디바이스 나침반/센서 기반 실시간 방위 측정** (제거).
- **방위(향)를 이용한 단지·매물 추천 또는 점수화** (제거 — 향은 단순 안내만).
- 3D 실내 공간 스캐닝/AR 배치 시뮬레이션.
- 부동산 매물 실거래·시세 중개 (외부 데이터 참조만).
- 전문 상담사 1:1 라이브 상담.
- 효능을 주장하는 의료/치유 콘텐츠.

## 5. 핵심 기능 요구사항 (Functional Requirements)

> 표기: **US**(User Story), **AC**(Acceptance Criteria), **Edge**(엣지케이스)

### 5.1 명당 조건 분석 엔진 (지명 & 지형 · 행정동 기준)
- **사주 오행 산출:** 생년월일시 입력 → 만세력 알고리즘으로 木·火·土·金·水 비중 계산.
- **지명 매칭 로직:** 부족 오행을 보완하는 한자가 포함된 **행정동명** DB 매칭.
  - Ex: 水 필요 → '水·川·江·海' 포함 행정동 리스트업.
- **지형 특성 매칭(행정동 단위):** 행정동별로 미리 정리한 지형 특성(고지대 / 수변 근접 / 평지 / 녹지 등)을 사주에 맞춰 안내·필터.
  - Ex: 水 보완 → 수변 인접 행정동 가중. 木 보완 → 녹지 비율 높은 행정동 가중.
  - 판정 기준은 **풍수 자문으로 단일 기준 확정**(§12 R4와 연계).
- **방위(향) 안내:** 사주 기준 길한 향(예: 남동향)을 **단순 참고 안내**로만 제공. 단지·매물 추천이나 점수화에는 사용하지 않는다.

**US:** 사용자는 생년월일시를 넣으면 내게 맞는 동네(행정동) 후보와 그 근거(오행·지형)를 보고 싶다.
**AC:**
- 추천 행정동마다 "왜 추천되는지" 한 줄 근거(매칭된 한자/오행/지형) 노출.
- 결과는 점수/순위가 아닌 **참고 카드**로 제시(효능 단정 금지).
- 지형/지명은 **사전 정리된 행정동 속성**을 조회만 하므로 런타임 부담이 낮도록 설계.

**Edge:**
- **출생 시(時) 모름:** "시 모름" 옵션 제공 → 시주 제외 분석으로 폴백, 정확도 한계 명시.
- **지형 속성 공백 행정동:** 속성이 없는 지역은 지형 가중을 빼고 지명 매칭만으로 폴백.
- 한자명이 없는 순우리말 동네 → 음/의미 매핑 또는 제외 규칙 정의.
- 동음이의 한자(예: 수=水/壽/秀) 오매칭 방지: DB에 정자(正字) 한자 필드 보유.

### 5.2 수면 풍수 가이드 (침대 머리 방향)
- **길방 산출:** 반안살/용신 등 사주 기준으로 머리 두기 좋은 방향(예: 동쪽) 도출.
- **텍스트/일러스트 안내:** "머리는 동쪽으로 두기를 권합니다" 식의 방위 안내를 카드로 제공. **별도 센서·나침반 없이 방위 기준 안내만** 제공.
- **대안 비보 팁:** 구조상 그 방향이 어려울 때 활용할 대체 인테리어 팁 텍스트.

**US:** 사용자는 침대 머리를 어느 쪽에 두면 좋을지 알고 싶다.
**AC:** 단정이 아닌 "권장" 톤으로 안내(효능 단정 금지). 대안 팁을 항상 함께 제공.

### 5.3 개운 인테리어 커머스 큐레이션
- **오행 기반 상품 태깅:** 상품 DB에 오행 속성(컬러·재질·형태) 태그.
- **개인화 피드:** 결과 화면 하단에 결핍 오행 보완 상품 피드(외부 제휴 아웃링크).

**AC:**
- 제휴 링크에는 **대가성 표기**(예: "쿠팡 파트너스 활동의 일정액 수수료" 등) 필수(§11).
- 품절/링크 만료 상품 비노출 처리.

### 5.4 데일리 운세 & 개운 루틴 *(리텐션 엔진)*
- **오늘의 일진/컬러:** 매일 갱신되는 가벼운 데일리 카드.
- **개운 미션:** "오늘은 동쪽에 작은 식물을 두기" 등 1일 1행동(→ 커머스 자연 연결).
- **푸시 알림:** 아침 시간대 데일리 카드 알림(옵트인, 빈도 조절 가능).

**AC:** 데일리 콘텐츠는 자정 기준 갱신. 알림은 옵트인이며 앱 내에서 끌 수 있어야 함.

### 5.5 프로필 & 궁합 *(Persona B 지원)*
- 본인 외 **파트너/가족 사주 프로필** 저장.
- 부부 합산 오행 기반 동네(행정동) 안내(§5.1 확장).

**Edge:** 프로필은 개인정보이므로 저장/삭제 정책을 §11과 정렬.

## 6. 정보 구조(IA) & 핵심 유저 플로우

### 6.1 온보딩 (첫 진입)
`스플래시 → 가치 소개 → 개인정보 수집 동의 → 생년월일시 입력(+'시 모름') → 오행 분석 로딩 → 결과 홈`

## 7. UI/UX 및 디자인 요구사항

- **툴:** Figma 기반 컴포넌트 디자인 시스템.
- **무드:** 무속색 배제, 미니멀·세련된 매거진 스타일. 톤다운 뉴트럴 + 자연물(물·돌·나무) 그래픽 모티프.
- **타이포그래피 / 폰트 세팅:**
  - 명리 데이터·운세 텍스트는 신뢰감 + 트렌디함을 동시에.
  - 영문/숫자 혼용 시 **Italic / Oblique 렌더링 안정화**: 가짜 기울임(faux italic) 깨짐 방지를 위해 이탤릭 서브패밀리 보유 폰트 채택 또는 폰트 메타데이터 정합성 검증을 디자인 QA에 포함.
  - 한글·한자·라틴·숫자 **폰트 폴백 체인** 정의.
- **마이크로 인터랙션:** 결과 로딩 시 오행 기운이 섞이는 부드러운 파티클 애니메이션.
- **접근성:** §10.3 참조.

## 10. 비기능 요구사항 (Non-Functional)

### 10.1 성능
- 결과 산출은 체감 지연 없이 수 초 내. 마지막 결과·데일리 카드는 캐시 조회 가능.

### 10.2 보안
- 생년월일시 등 개인정보 **전송 구간 암호화(TLS) + 저장 시 암호화**.
- 가능하면 오행 계산을 **클라이언트 단에서 수행**해 원시 개인정보의 서버 전송 최소화.

### 10.3 접근성
- 명암비 WCAG AA, 동적 글자 크기 대응, 그래픽 요소에 텍스트 대체·스크린리더 레이블 제공.

### 10.4 플랫폼
- iOS / Android.

## 14. 데이터 / API / 기술

### 14.1 명리 데이터
- 오픈소스 만세력 라이브러리(JS/Python) + 풍수 자문 매핑 테이블.

> ⚠️ **정확도 주의:** 출생 시각 보정(진태양시·과거 표준시/서머타임 변천)과 자시 경계·윤달 처리가 사주 결과를 좌우한다. **채택 기준을 하나로 고정**하고 전문가·공신력 자료로 검증할 것(현재 환경에서 외부 검증 불가).

### 14.2 행정구역 DB
- 공공데이터포털 전국 **행정동** DB(한자명 포함). 매칭·표기 단위는 행정동 기준. 한자 정자 필드·갱신 주기 정의.

### 14.3 지형 데이터 (행정동 단위)
- 공공 GIS(고도·수계·녹지 등)에서 **행정동별 지형 특성**(고지대/수변 근접/평지/녹지)을 **사전 정리해 속성 테이블화**.
- 오프라인 사전 계산 → 런타임은 **조회만**. 판정 임계값만 풍수 자문으로 확정.

### 14.4 아키텍처 메모
- 개인정보 보호를 위해 **오행 계산을 가능한 클라이언트에서 수행**하고, 서버는 매핑 테이블·상품 DB·콘텐츠 제공 중심으로 설계.

## 18. 부록 — 용어
